# ADR 0001. Классификатор транспортных средств (виды и типы)

- Статус: Принято
- Этап: 1 (основа справочника ТС)
- Связано: `apps/api/drizzle/0009_vehicle_classification.sql`, `apps/api/src/db/schema.ts`, `docs/database-schema.md`

## Контекст

Нужен справочник ТС механизации: два верхнеуровневых вида — «Спецтехника» и
«Грузоперевозки» — и иерархия типов/подтипов под ними (напр. «Экскаваторы» →
«Экскаваторы гусеничные»). Конкретные ТС (следующие этапы) будут ссылаться на
подтип. Существующий `container_types` описывает контейнеры/самосвалы для вывоза
мусора и к парку механизации отношения не имеет.

## Решение

1. **Виды — таблицей `vehicle_kinds`, а не PostgreSQL enum.** Добавление нового вида
   не требует `ALTER TYPE`/миграции типа; вид — управляемый справочник
   (`code`, `name`, `sort_order`, `is_active`), по соглашениям `container_types`.
2. **Типы и подтипы — в ОДНОЙ иерархической таблице `vehicle_types`** (`parent_id`),
   а не двумя таблицами `vehicle_types`/`vehicle_subtypes`. Единая модель, проще
   запросы; уровень вычисляется по `parent_id`, поле `level` не храним.
3. **Принадлежность к виду — на уровне БД.** `kind_id` денормализован на каждой
   строке; составной FK `(parent_id, kind_id) → vehicle_types(id, kind_id)` (требует
   `UNIQUE(id, kind_id)`) гарантирует, что родитель и дочерний тип одного вида —
   надёжнее проверки в приложении. При `parent_id IS NULL` (MATCH SIMPLE) FK не
   проверяется — верхний уровень допускается.
4. **Глубина — строго 2 уровня** (тип → подтип). БД жёстко её не ограничивает (только
   `CHECK parent_id <> id` от самоссылки); запрет «внуков» и циклов — ответственность
   приложения.
5. **Назначаемость — `is_selectable`.** Родительский тип — `false`, конечный подтип —
   `true`. «Конкретное ТС ссылается только на selectable» на этапе 1 — конвенция
   приложения; жёсткий FK/ограничение решим на этапе с самой техникой.
6. **Не сливаем с `container_types`.** Разные домены (вывоз мусора vs парк
   механизации); у `container_types` свой дискриминатор `type` (`cont`/`truck`).
7. **persons/drivers/документы сотрудников — вне этого этапа.** Здесь только
   классификатор ТС.

## Последствия

- (+) Виды расширяются без изменения типов БД.
- (+) Целостность «одна ветка — один вид» гарантируется БД, не только кодом.
- (−) `kind_id` дублируется на подтипах — поддерживается составным FK.
- (−) Глубину > 2 и циклы БД не ловит — контролирует приложение (см. п.4).
- Удаление используемого вида/типа блокируется `ON DELETE RESTRICT`.
