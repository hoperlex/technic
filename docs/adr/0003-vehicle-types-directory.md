# ADR 0003. Справочник «Типы/подтипы ТС» — вертикальный срез (этап 2.1)

- Статус: Принято
- Этап: 2.1 (CRUD-справочник поверх наполненного классификатора)
- Основано на: [ADR 0001](0001-vehicle-classification.md), [ADR 0002](0002-vehicle-classification-population.md)
- Связано: `packages/contracts/src/vehicle-types.ts`, `apps/api/src/routes/vehicle-types.ts`,
  `apps/api/src/routes/vehicle-kinds.ts`, `apps/api/drizzle/0011_vehicle_types_invariants.sql`,
  `apps/web/src/pages/directories/VehicleTypesTab.tsx`

## Контекст

После наполнения классификатора (этап 2) нужен законченный срез БД → контракты → REST →
frontend для просмотра и редактирования типов/подтипов ТС. Ранее обсуждавшееся ограничение
«тип с подтипами не редактируется» признано избыточным: наличие подтипов должно защищать
**структуру и ссылки**, но не мешать исправить наименование/описание.

## Решения

1. **Уровень через `level`.** Создание — `z.discriminatedUnion('level', [createParentType, createSubtype])`.
   Тип: клиент шлёт `kindId`, backend ставит `parentId=null, isSelectable=false, isActive=false`.
   Подтип: клиент шлёт `parentId` (не `kindId`), backend берёт `kindId` из родителя,
   ставит `isSelectable=true`. Строгие (`.strict()`) схемы отклоняют лишние/структурные поля.
2. **Редактирование типа не блокируется подтипами.** PATCH типа меняет только
   `name/description/sortOrder` (`updateParentTypeSchema`). Наличие активных/неактивных
   подтипов не влияет на возможность правки описательных полей.
3. **Структурные поля неизменяемы.** Для типа и подтипа нельзя менять
   `id/code/kindId/parentId/level/isSelectable`; для подтипа `parentId` удалён из update-схемы —
   перенос между типами/видами через PATCH невозможен (защита будущих ссылок
   `vehicles.vehicle_type_id`, моделей, source-mappings от скрытой смены смысла).
4. **Удаления нет.** Нет `DELETE`/`restore`/`deleted_at`. Тип не удаляется; подтип
   деактивируется через `PATCH isActive=false`. Неактивный подтип остаётся в справочнике,
   сохраняет ссылки, может быть реактивирован.
5. **Активность типа — производная.** `parent.isActive = EXISTS(active child)`; пересчёт после
   создания/активации/деактивации подтипа. Клиент не передаёт активность типа. Так как
   `parentId` неизменяем, пересчёт при переносе не нужен.
6. **Инварианты в БД (миграция 0011).** CHECK согласованности `parent_id`/`is_selectable`
   и CHECK формата кода `^[a-z][a-z0-9_]*$`. Все данные сида 0010 им удовлетворяют.
7. **HTTP-коды.** 400 — базовый Zod / запрещённое поле; 404 — тип/родитель не найден;
   409 — дублирующий код; 422 — родителем выбран подтип / попытка третьего уровня
   (`err.unprocessable`).
8. **Транзакции.** Создание подтипа и PATCH — в транзакции с `SELECT … FOR UPDATE`
   родителя/записи и последующим пересчётом активности.
9. **Список.** Режимы `view=list|hierarchy`; иерархия сортирует
   `kind.sortOrder → parent.sortOrder → parent.name → child.sortOrder → child.name`,
   родитель и его подтипы идут подряд. Frontend грузит `view=hierarchy&pageSize=500`.
10. **Frontend.** Единственная вкладка «Справочники → Типы/подтипы ТС». Плоская таблица,
    колонки строго **Вид → Тип → Подтип → Активен → Действия**: у строки типа Подтип = «—»,
    у строки подтипа Тип = `parentName`. Активность подтипа — инлайн-`Switch`
    (деактивация с подтверждением); у типа — readonly-бейдж. Виды ТС — только чтение
    (источник Select/фильтра), управление видами на этапе не реализуется.
11. **Выравнивание container-types.** Справочник контейнеров приведён к той же модели:
    убран `DELETE` (деактивация через `PATCH isActive`), strict update-схема без `code`,
    `code` readonly в форме, инлайн-`Switch` активности.

## Последствия

- (+) Структура и ссылки защищены; обычное переименование не блокируется.
- (+) Два справочника (ТС и контейнеры) ведут себя единообразно.
- (−) Перенос подтипа = создать новый + деактивировать старый (перенос связанных ТС —
  отдельной процедурой на следующем этапе).
- (−) Иерархический режим предполагает, что `pageSize` покрывает набор (сейчас ≤ 500 строк);
  постраничная разбивка семей не реализована — не требуется на текущем объёме.
