# ADR 0002. Наполнение классификатора ТС и source mappings (этап 2)

- Статус: Принято
- Этап: 2 (наполнение типов/подтипов)
- Основано на: [ADR 0001](0001-vehicle-classification.md); два Excel (полный реестр + работавшая техника)
- Связано: `apps/api/drizzle/0010_vehicle_types_seed.sql`, `apps/api/src/db/vehicle-classification-data.ts`, `apps/api/src/scripts/analyze-vehicle-types.ts`

## Контекст

Типизация проведена по двум источникам:

- **Полный реестр** «Список автотехники» (лист `Лист_1`) — марка, модель, госномер, фактический подтип; авторитетен при расхождении.
- **Список работавшей техники**, лист `06.26-06.27` (**94 экземпляра**) — признак активности. Лист `01.25-07.26` для активности не используется.

## Решения

1. **Приоритет полного реестра.** При расхождении исходного «Тип ТС» и реестра фактический подтип берётся из реестра.
2. **Активность.** Подтип `is_active=true`, если на активном листе есть ≥1 его экземпляр, иначе `false`. Родитель активен ⇔ есть активный потомок. Оба вида остаются активными.
3. **Полное исключение самосвалов.** `dump_truck`/«Самосвал» не создаются; один самосвал с активного листа исключается до расчёта (94 → 93); mapping «Самосвалы» не создаётся; `bunker_carrier` не создаётся.
4. **Source mappings** (`vehicle_type_source_mappings`). Однозначные «Тип ТС» → конечный подтип (`direct`); неоднозначные → **родитель** (`is_selectable=false`) с `requires_instance_resolution=true` (`by_model`/`by_registry`). Укрупнённая группа не назначается конкретному ТС напрямую.
5. **Ручные исправления** (по реестру): `JCB TLT30C` → `forklift`; `МАЗ-МКМВ9В У702/У782` → `multilift`; `ГАЗ-33106` с КМУ → `light_kmu_truck`; `JAC` с КМУ → `light_kmu_truck`.
6. **`is_active` — снимок** на дату анализа: конкретные ТС на этом этапе в БД не хранятся; пересчёт активности — при импорте ТС (следующий этап).
7. **Контрольные итоги.** Строк 94; исключён 1; классифицировано **93** (спецтехника 65 + грузоперевозки 28); подтипов **22** (16 активных + 6 неактивных).

## Неоднозначные исходные группы

`Вилочные погрузчики и минипогрузчики` → `loaders`; `Грузовые малотоннажные автомобили` → `road_transport` (часть — `light_kmu_truck` по реестру); `Тяжелые манипуляторы` → `kmu_transport`; `Бункеровозы, мультилифты` → `specialized_transport`; `Экскаваторы-погрузчики` → `excavators`.

## Повторный анализ при обновлении Excel

Запустить `apps/api/src/scripts/analyze-vehicle-types.ts <реестр.xlsx> <работавшая.xlsx>`; сверить counts с `vehicle-classification-data.ts`; при расхождении — обновить константы и добавить **новую** сид-миграцию (старые не редактировать).

## Последствия

- (+) Классификация воспроизводима: mappings + overrides + анализатор.
- (−) Неоднозначные группы требуют per-instance разбора (`by_model`/`by_registry`) — на этапе импорта ТС.
- (−) `is_active` устаревает без пересчёта при изменении парка.
